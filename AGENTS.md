# Repository Agent Guidelines

本文件适用于整个仓库，是所有进入本仓库工作的 Codex 或 Agent 的首要执行入口。更深目录如果未来存在自己的 `AGENTS.md`，在对应目录内优先遵循更具体的规则。

产品方向、当前阶段和协作机制分别以以下文档为准：

- `docs/product/positioning.md`
- `docs/product/current-stage.md`
- `docs/ai/operating-system.md`

不要在本文件中重复完整产品定义。上述文档存在冲突时，停止扩大任务范围，向产品负责人指出冲突并等待确认。

## 当前项目基线

- 当前产品是面向中文个人投资者的个人资产追踪 Web MVP。
- 当前阶段是“上线准备阶段（种子用户试用版本准备）”。
- 当前唯一目标是冻结并验收一个让目标用户独立建立首个清晰、可信整体资产视图的种子试用版本。
- 当前优先级高于新增功能：私人资产数据边界一致、本地数据安全、稳定版本冻结、核心路径验收、种子用户测试准备。
- 已实现能力不等于已通过真实用户验证；未提交代码不等于稳定能力。

长期需要但不属于当前门槛：完整云端同步、多设备一致性、实时和全市场行情、完整社区、订阅支付、大规模架构迁移、高级归因和完整复盘体系。

## 开始任务前的最低必读

### 产品定位、阶段或 Roadmap 任务

必须先读：

- `docs/product/positioning.md`
- `docs/product/current-stage.md`
- `docs/ai/operating-system.md`

### 产品方案、流程或 UI/UX 任务

必须先读：

- 上述三份基线文档；
- 相关 PRD、`docs/decisions/` 中的决策记录和现有实现；
- 必要时查看当前部署或截图。

不得只根据旧计划文档重做产品判断。

### 代码开发任务

必须先读：

- 当前任务说明；较大任务参考 `docs/ai/task-template.md`；
- 相关代码和测试；
- 涉及产品逻辑时同时读取产品定位和当前阶段文档；
- 涉及数据、隐私、行情或计算时读取相关决策与 `docs/data/` 下的 API 或数据文档。

### 用户测试、发布或 QA 任务

必须先读：

- `docs/product/current-stage.md`
- `docs/product/launch-smoke-test.md`
- `docs/product/seed-user-trial.md`
- 相关发布与回滚说明。

如果对应文件不存在或仍为空，明确报告，不自行假设内容。

## 任务分类与处理方式

### 1. 只读分析任务

适用于仓库审查、产品评审、技术风险检查、数据流审计、测试覆盖检查和阶段判断。

- 未经明确要求不得修改任何文件。
- 输出事实、证据、推断和无法确认项。
- 代码已存在不代表产品能力已验证。
- 未提交代码不代表稳定能力。

### 2. 文档写入任务

- 只修改明确指定的文件，不顺带更新相邻文档。
- 保留已确认结论，不擅自新增产品方向。
- 完成后展示完整 diff，并说明是否存在未跟踪文件。
- 不自动暂存或提交。

### 3. 产品开发任务

- 先确认目标、范围、非范围和验收标准；较大任务先输出实施计划。
- 不因发现邻近问题就自动扩大范围。
- 优先复用现有结构。
- 只有在技术债影响当前验证、正确性或维护时才处理。
- 完成后运行必要测试并按 `docs/ai/handoff-template.md` 交接。

### 4. Bug 修复任务

- 先复现或定位并说明根因。
- 使用最小安全修复，增加或更新必要测试。
- 金融计算、权限、安全或隐私 Bug 必须先补充或更新可复现测试。
- 不借 Bug 修复进行无关重构。

### 5. 高风险任务

高风险范围包括数据存储与上传、用户资产和交易数据、身份认证、数据库迁移、金融计算、行情来源和缓存、导入导出与恢复、生产部署、隐私和对外承诺。

- 修改前必须检查真实数据流，不能只依赖文档描述。
- 产品文案与代码行为必须一致。
- 不得静默吞掉数据错误。
- 不得把缺失、过期、手动或估算数据表达为正常真实数据。
- 必须说明回滚方案和数据影响。
- 未经批准不得执行生产高风险操作。

## 产品决策边界

Codex 或 Agent 不得自行决定：

- 修改产品定位、目标用户或 MVP 第一价值；
- 扩大 MVP 核心范围；
- 将长期能力提升为当前门槛；
- 修改私人数据存储和上传承诺；
- 修改社区公开范围；
- 输出个性化投资建议；
- 判断进入下一产品阶段；
- 修改重大产品决策记录。

遇到更优方案时可以提出，但必须说明原因并标记为建议，未经确认不得直接实现。

## 当前产品与数据原则

- 产品用于资产记录和分析，不构成投资建议，不得承诺收益、暗示保本、引导跟单或输出个性化买卖建议。
- 用户资产、交易、身份和笔记属于敏感数据，默认最小权限，不得使用真实隐私数据作为代码或测试样例。
- 公共行情服务与私人资产云同步是两件不同的事。
- 当前私人资产数据以本地保存为主，但实际数据流必须以代码审计结果为准。
- 价格、净值和汇率应标明来源、时间和状态；缺失、过期、手动和估算状态必须明确。
- 金融计算必须可解释、可复算、可测试；金额、数量、价格、费率和收益率不得使用会明显影响结果的浮点方案。
- 金融逻辑不得散落在 UI 中；字段应明确单位、币种、精度和时间语义。
- 数据损坏不得静默回退成正常 Demo 数据。
- 导入和恢复必须有校验、预览和确认；导出、删除和恢复应避免不可逆误操作。
- 用户数据不得默认公开；社区观点与私人资产数据必须隔离。

## 代码与架构规则

当前是 npm workspace monorepo，workspaces 为 `apps/*` 和 `packages/*`：

- 根目录 `index.html`、`src/`、`server.mjs`：当前原生 JavaScript 静态 MVP 主体验及本地预览入口。
- `apps/web/`：并行开发的 React + TypeScript 入口，不得默认视为正式种子试用入口。
- `apps/api/`：行情、资源搜索及相关服务端 API。
- `packages/domain/`：React/TypeScript 方向的共享领域类型与计算包。
- `src/domain/`：当前原生 MVP 的金融计算和市场数据边界。
- `db/`：数据库 schema；`scripts/market-data/`：行情、资源库、迁移和回补脚本。
- `tests/unit/`、`tests/integration/`：现有 Node 单元与集成测试。

执行规则：

- 新旧应用并行时，任务必须明确作用于哪个入口；不得把两个入口都默认当成正式主体验。
- 不要在 `src/domain/`、`packages/domain/`、UI 或 API 中随意复制领域逻辑。
- 金融计算、价格状态和数据校验优先放在可测试的领域层或服务边界。
- 前后端 API 变更必须同步检查 `docs/data/api-contract.md`、调用方、错误状态和兼容性。
- 数据库变更必须考虑 migration、事务、回滚、已有数据兼容和审计影响。
- 外部行情与汇率通过适配边界接入，保留来源、拉取时间、失败状态和降级策略。
- 不要为了“更现代”进行大规模 React/TypeScript 或架构迁移。
- 不提交构建产物、真实密钥、真实账户数据或本地环境文件；`apps/web/dist/` 等生成内容只在任务明确要求时处理。
- 引入或改变工具后，按任务范围更新 `README.md` 的安装、启动、构建、测试和环境变量说明。

## 测试要求

只使用仓库已定义的命令：

- `npm test`：运行现有 Node 单元和集成测试。
- `npm run typecheck`：检查共享领域包和 React Web 入口类型。
- `npm run build`：类型检查并构建 React Web 入口。
- `npm run build:static`：构建当前原生 JS 静态入口。
- `npm start`：本地预览原生 JS 入口，默认 `http://localhost:4173`。
- `npm run dev`：运行 React 入口，默认 `http://localhost:5173`。
- `npm run api:start`：运行 API，默认健康检查为 `http://localhost:4180/api/health`。

按改动范围执行：

- 修改领域计算时运行相关单元测试，覆盖正常路径、精度边界、零值、负数、多币种、费用和复杂交易场景。
- 修改 API 时运行相关集成或 smoke 测试，并检查契约、鉴权、校验失败和回滚路径。
- 修改 UI 主流程时进行浏览器级检查；响应式页面检查 390px、768px 和 1280px。
- 涉及导入恢复时覆盖错误文件、取消、预览和确认。
- 涉及行情时覆盖成功、缺失、过期、失败和手动兜底。
- 涉及本地存储时覆盖解析失败、错误告知和数据恢复，禁止静默恢复 Demo。
- 无法运行某项测试时必须说明原因。
- 测试通过不等于产品已通过真实用户验证。

## UI 与体验要求

- UI 优化首先服务于理解、任务完成和可信感。
- 保持清晰的信息层级、克制的颜色、稳定的间距和适合工具使用的信息密度。
- 每个页面的主操作和下一步必须明确，避免无关按钮和装饰组件。
- 覆盖空状态、加载、错误、缺失、禁用、确认和恢复状态；重要语义不能只靠颜色表达。
- 图表必须解释组合价值、分布、趋势或归因，不添加装饰性图表。
- 当前阶段优先保证核心路径可用，不追求成熟商业产品的全部精度。
- UI 评审发现更优流程时可以提出，但未经确认不得改动核心产品逻辑。

## Git 与变更规则

- 开始任务前检查 Git 状态。
- 不覆盖用户已有未提交修改，不把未提交工作区当作稳定基线。
- 只修改任务范围内文件，不修改无关格式。
- 未经明确要求不暂存、不提交、不推送，不自动合并主分支。
- 大任务保持可审查 diff；发现范围外问题应报告，不顺手修改。
- 完成后列出修改文件、测试结果、未运行测试和遗留风险。

## 文档更新规则

- 周度结论写入 `docs/product/weekly-reviews/`。
- 当前阶段写入 `docs/product/current-stage.md`。
- 产品定位写入 `docs/product/positioning.md`。
- 重大决策写入 `docs/decisions/`。
- 产品变更写入相关 changelog。
- 普通代码任务不得自动修改定位、阶段或协作机制文档。
- 文档内容必须明确区分：已确认、根据仓库判断、待确认、状态不明确。
- 高风险产品、数据和阶段文档必须经过产品负责人明确批准后修改。

## 任务完成交接

每项修改任务最终至少说明：

1. 做了什么；
2. 用户可感知的变化；
3. 修改了哪些文件；
4. 运行了哪些测试及结果；
5. 哪些测试没有运行；
6. 已知风险；
7. 未完成或暂缓事项；
8. 是否需要更新文档；
9. 是否需要产品负责人确认；
10. Git 状态和是否存在未跟踪文件。

较大任务开始前参考 `docs/ai/task-template.md`，完成后参考 `docs/ai/handoff-template.md`。

## 人工审批操作

未经产品负责人确认不得执行：

- 生产数据库迁移；
- 删除、覆盖或修改真实用户数据；
- 生产发布；
- 自动合并主分支；
- 修改环境密钥；
- 改变数据存储和隐私承诺；
- 公开用户资产数据；
- 大规模架构迁移；
- 创建超出当前阶段范围的大功能。

## 最后检查

开始前：

- 读对文档了吗？
- 明确任务范围和非范围了吗？
- 检查 Git 状态了吗？
- 是否涉及敏感数据或高风险操作？

完成前：

- 是否只改了范围内内容？
- 是否运行了必要测试？
- 产品说明与代码行为是否一致？
- 是否留下可审查 diff？
- 是否完成交接说明？
