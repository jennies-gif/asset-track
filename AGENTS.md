# Repository Agent Guidelines

本文件适用于整个仓库，是所有进入本仓库工作的 Codex 或 Agent 的首要执行入口。更深目录如果未来存在自己的 `AGENTS.md`，在对应目录内优先遵循更具体的规则。

产品方向、当前阶段、协作机制和工程执行规范分别以以下文档为准：

- `docs/product/positioning.md`
- `docs/product/current-stage.md`
- `docs/ai/operating-system.md`
- `docs/engineering/architecture.md`
- `docs/engineering/coding-guidelines.md`
- `docs/engineering/design-system.md`

`AGENTS.md` 是 AI 的仓库执行入口，不是产品或工程百科。不要在本文件中重复完整产品定义、架构说明、编码细则或设计 Token。

规范优先级如下：

1. 用户明确指令；
2. `docs/product/positioning.md`；
3. `docs/product/current-stage.md`；
4. `docs/decisions/` 中已批准的 Decision Log；
5. `docs/engineering/architecture.md`；
6. `docs/engineering/coding-guidelines.md`；
7. `docs/engineering/design-system.md`；
8. 当前代码实现。

Markdown 用户明确指令代表当前任务执行意图。

但涉及以下事项时，不自动视为产品负责人批准：

- 产品定位变化；
- 当前阶段变化；
- 用户隐私边界变化；
- 私人数据上传；
- 生产真实数据修改；
- 对外产品承诺变化。

上述事项需要明确确认。

`docs/ai/operating-system.md` 负责协作、审批和写入流程，不用于覆盖产品定位、当前阶段或已批准决策。当前代码实现用于确认真实行为，不因优先级最低而被文档推断替代；文档与代码不一致时，必须报告差异，不能把文档描述当作已实现事实。规范之间存在冲突时，停止扩大任务范围，向产品负责人指出冲突并等待确认。

## 当前项目基线

- 当前产品是面向中文个人投资者的个人资产追踪 Web MVP。
- 当前阶段是“上线准备阶段（种子用户试用版本准备）”。
- 当前唯一目标是冻结并验收一个让目标用户独立建立首个清晰、可信整体资产视图的种子试用版本。
- 当前优先级高于新增功能：私人资产数据边界一致、本地数据安全、稳定版本冻结、核心路径验收、种子用户测试准备。
- 已实现能力不等于已通过真实用户验证；未提交代码不等于稳定能力。

长期需要但不属于当前门槛：完整云端同步、多设备一致性、实时和全市场行情、完整社区、订阅支付、大规模架构迁移、高级归因和完整复盘体系。

## 开始任务前的最低必读

### 产品定位、阶段或 Roadmap 任务

必须先读：

- `docs/product/positioning.md`
- `docs/product/current-stage.md`
- `docs/ai/operating-system.md`

### 产品方案、流程或 UI/UX 任务

必须先读：

- 上述三份基线文档；
- 相关 PRD、`docs/decisions/` 中的决策记录和现有实现；
- 必要时查看当前部署或截图。

不得只根据旧计划文档重做产品判断。

### 代码开发任务

必须先读：

- 当前任务说明；较大任务参考 `docs/ai/task-template.md`；
- 相关代码和测试；
- 所有代码开发、Bug 修复和代码评审读取 `docs/engineering/coding-guidelines.md`；
- 涉及新功能、跨模块修改、核心业务逻辑、领域模型、API、数据库、持久化、同步或架构重构时，同时读取 `docs/engineering/architecture.md`；
- 涉及 UI、CSS、组件、图表、响应式或视觉重构时，同时读取 `docs/engineering/design-system.md`，并检查现有 Token、组件和样式实现；
- 涉及产品逻辑时同时读取产品定位和当前阶段文档；
- 涉及数据、隐私、行情或计算时读取相关决策与 `docs/data/` 下的 API 或数据文档。

只读取与任务影响面相关的工程文档，不要求每个任务读取全部三份。工程文档不能替代对现有代码、组件、数据流和测试的检查。

### 用户测试、发布或 QA 任务

必须先读：

- `docs/product/current-stage.md`
- `docs/product/launch-smoke-test.md`
- `docs/product/seed-user-trial.md`
- 相关发布与回滚说明。

如果对应文件不存在或仍为空，明确报告，不自行假设内容。

## 任务分类与处理方式

### 1. 只读分析任务

适用于仓库审查、产品评审、技术风险检查、数据流审计、测试覆盖检查和阶段判断。

- 未经明确要求不得修改任何文件。
- 输出事实、证据、推断和无法确认项。
- 代码已存在不代表产品能力已验证。
- 未提交代码不代表稳定能力。

### 2. 文档写入任务

- 只修改明确指定的文件，不顺带更新相邻文档。
- 保留已确认结论，不擅自新增产品方向。
- 完成后展示完整 diff，并说明是否存在未跟踪文件。
- 不自动暂存或提交。

### 3. 产品开发任务

- 先确认目标、范围、非范围和验收标准；较大任务先输出实施计划。
- 新功能、跨层变更和高风险修改在实施前必须完成与风险相称的工程判断，至少说明：
  - 修改作用于原生 JavaScript 主体验候选入口、React 入口、API、共享领域层或多个入口；
  - 数据类型、数据归属、当前事实来源、保存位置及是否发送到服务器；
  - 是否涉及私人或敏感数据，是否改变真实存储或传输边界；
  - 业务逻辑应进入哪个模块或层，是否已有可复用实现，是否会形成跨入口或跨模块重复；
  - 是否影响持久化 schema、版本、校验、迁移、导入导出和恢复；
  - 临时方案的限制、技术债、迁移方向和触发条件；
  - 是否触发 Architecture Decision，以及测试范围、数据影响和回滚方式。
- 微小、孤立、低风险修改可以简要完成上述判断；不得因工程判断自动扩大任务范围。
- 不因发现邻近问题就自动扩大范围。
- 优先复用现有结构。
- 只有在技术债影响当前验证、正确性或维护时才处理。
- 完成后运行必要测试并按 `docs/ai/handoff-template.md` 交接。

### 4. Bug 修复任务

- 先复现或定位并说明根因。
- 使用最小安全修复，增加或更新必要测试。
- 金融计算、权限、安全或隐私 Bug 必须先补充或更新可复现测试。
- 不借 Bug 修复进行无关重构。

### 5. 高风险任务

高风险范围包括数据存储与上传、用户资产和交易数据、身份认证、数据库迁移、金融计算、行情来源和缓存、导入导出与恢复、生产部署、隐私和对外承诺。

- 修改前必须检查真实数据流，不能只依赖文档描述。
- 产品文案与代码行为必须一致。
- 不得静默吞掉数据错误。
- 不得把缺失、过期、手动或估算数据表达为正常真实数据。
- 必须说明回滚方案和数据影响。
- 未经批准不得执行生产高风险操作。

## 产品决策边界

Codex 或 Agent 不得自行决定：

- 修改产品定位、目标用户或 MVP 第一价值；
- 扩大 MVP 核心范围；
- 将长期能力提升为当前门槛；
- 修改私人数据存储和上传承诺；
- 修改社区公开范围；
- 输出个性化投资建议；
- 判断进入下一产品阶段；
- 修改重大产品决策记录。

遇到更优方案时可以提出，但必须说明原因并标记为建议，未经确认不得直接实现。

## 当前产品与数据原则

- 产品用于资产记录和分析，不构成投资建议，不得承诺收益、暗示保本、引导跟单或输出个性化买卖建议。
- 用户资产、交易、身份和笔记属于敏感数据，默认最小权限，不得使用真实隐私数据作为代码或测试样例。
- 公共行情服务与私人资产云同步是两件不同的事。
- 当前私人资产数据以本地保存为主，但实际数据流必须以代码审计结果为准。
- 价格、净值和汇率应标明来源、时间和状态；缺失、过期、手动和估算状态必须明确。
- 金融计算必须可解释、可复算、可测试；金额、数量、价格、费率和收益率不得使用会明显影响结果的浮点方案。
- 金融逻辑不得散落在 UI 中；字段应明确单位、币种、精度和时间语义。
- 数据损坏不得静默回退成正常 Demo 数据。
- 导入和恢复必须有校验、预览和确认；导出、删除和恢复应避免不可逆误操作。
- 用户数据不得默认公开；社区观点与私人资产数据必须隔离。

## 代码与架构规则

当前是 npm workspace monorepo，workspaces 为 `apps/*` 和 `packages/*`：

- 根目录 `index.html`、`src/`、`server.mjs`：当前原生 JavaScript 静态 MVP 是种子用户试用版本的主体验候选入口。
- `apps/web/`：React + TypeScript 是长期演进方向，不作为当前阶段默认替换方案。
- `apps/api/`：行情、资源搜索及相关服务端 API。
- `packages/domain/`：React/TypeScript 方向的共享领域类型与计算包。
- `src/domain/`：当前原生 MVP 的金融计算和市场数据边界。
- `db/`：数据库 schema；`scripts/market-data/`：行情、资源库、迁移和回补脚本。
- `tests/unit/`、`tests/integration/`：现有 Node 单元与集成测试。

执行规则：

- 新旧应用并行时，任务必须明确作用于哪个入口；不得把两个入口都默认当成正式主体验。
- 一个业务规则只能有一个可信实现；不得在 `src/domain/`、`packages/domain/`、UI、API 或两个前端入口中复制领域逻辑。
- 核心领域和金融逻辑不得散落在页面、Render、API 路由或多个入口；金融计算、价格状态和数据校验优先放在可测试的领域层或服务边界。
- Domain 不直接访问 DOM、API、`localStorage`、数据库或 UI；外部能力和副作用通过明确的 service、API 或 data 边界管理。
- 持久化业务数据必须考虑 schema、版本、校验、迁移、错误处理和恢复，不得以无结构数据或静默回退掩盖损坏。
- 前后端 API 变更必须同步检查 `docs/data/api-contract.md`、调用方、错误状态和兼容性。
- 数据库变更必须考虑 migration、事务、回滚、已有数据兼容和审计影响。
- 外部行情与汇率通过适配边界接入，保留来源、拉取时间、失败状态和降级策略。
- 新 UI 优先复用现有 Token、组件和状态语义；不得新增长期覆盖型 CSS、页面专用视觉体系或重复组件，重要状态不得只依赖颜色表达。
- 旧代码按当前风险和价值渐进收敛；不得仅因与长期目标架构或设计系统不完全一致而自动重构。
- 临时实现必须有明确目的、生命周期和迁移方向，不得通过无边界补丁扩大技术债。
- 不要为了“更现代”进行大规模 React/TypeScript 或架构迁移。
- 不提交构建产物、真实密钥、真实账户数据或本地环境文件；`apps/web/dist/` 等生成内容只在任务明确要求时处理。
- 引入或改变工具后，按任务范围更新 `README.md` 的安装、启动、构建、测试和环境变量说明。

## 测试要求

只使用仓库已定义的命令：

- `npm test`：运行现有 Node 单元和集成测试。
- `npm run typecheck`：检查共享领域包和 React Web 入口类型。
- `npm run build`：类型检查并构建 React Web 入口。
- `npm run build:static`：构建当前原生 JS 静态入口。
- `npm start`：本地预览原生 JS 入口，默认 `http://localhost:4173`。
- `npm run dev`：运行 React 入口，默认 `http://localhost:5173`。
- `npm run api:start`：运行 API，默认健康检查为 `http://localhost:4180/api/health`。

按改动范围执行：

- 修改领域计算时运行相关单元测试，覆盖正常路径、精度边界、零值、负数、多币种、费用和复杂交易场景。
- 修改 API 时运行相关集成或 smoke 测试，并检查契约、鉴权、校验失败和回滚路径。
- 修改 UI 主流程时进行浏览器级检查；响应式页面检查 390px、768px 和 1280px。
- 涉及导入恢复时覆盖错误文件、取消、预览和确认。
- 涉及行情时覆盖成功、缺失、过期、失败和手动兜底。
- 涉及本地存储时覆盖解析失败、错误告知和数据恢复，禁止静默恢复 Demo。
- 无法运行某项测试时必须说明原因。
- 测试通过不等于产品已通过真实用户验证。

## UI 与体验要求

- UI 优化首先服务于理解、任务完成和可信感。
- 保持清晰的信息层级、克制的颜色、稳定的间距和适合工具使用的信息密度。
- 每个页面的主操作和下一步必须明确，避免无关按钮和装饰组件。
- 覆盖空状态、加载、错误、缺失、禁用、确认和恢复状态；重要语义不能只靠颜色表达。
- 图表必须解释组合价值、分布、趋势或归因，不添加装饰性图表。
- 当前阶段优先保证核心路径可用，不追求成熟商业产品的全部精度。
- UI 评审发现更优流程时可以提出，但未经确认不得改动核心产品逻辑。

## Git 与变更规则

- 开始任务前检查 Git 状态。
- 不覆盖用户已有未提交修改，不把未提交工作区当作稳定基线。
- 只修改任务范围内文件，不修改无关格式。
- 未经明确要求不暂存、不提交、不推送，不自动合并主分支。
- 大任务保持可审查 diff；发现范围外问题应报告，不顺手修改。
- 完成后列出修改文件、测试结果、未运行测试和遗留风险。

## 文档更新规则

- 周度结论写入 `docs/product/weekly-reviews/`。
- 当前阶段写入 `docs/product/current-stage.md`。
- 产品定位写入 `docs/product/positioning.md`。
- 重大决策写入 `docs/decisions/`。
- 产品变更写入相关 changelog。
- 普通代码任务不得自动修改定位、阶段或协作机制文档。
- 文档内容必须明确区分：已确认、根据仓库判断、待确认、状态不明确。
- 高风险产品、数据和阶段文档必须经过产品负责人明确批准后修改。

### Architecture Decision 权限与状态

Agent 必须识别并报告 `docs/engineering/architecture.md` 定义的 Architecture Decision 触发条件，但不得借此自动扩大代码或文档范围。

ADR 使用以下状态：

- `Not Required`：已检查，不触发 ADR；
- `Proposed`：已识别决策需求，尚未形成获授权草案；
- `Draft`：已获文档写入授权并形成草案，尚未批准；
- `Accepted`：已由有权负责人批准并生效；
- `Deprecated`：曾经生效，现已被替代或不再适用，必须保留替代关系或原因。

权限规则：

- 未经明确的文档写入授权，不得创建、修改或改变 ADR 状态；未获授权时只在计划或交接中报告建议标题、触发原因、待决策项以及数据、隐私、兼容、迁移和回滚影响，状态为 `Proposed`；
- 已授权的开发任务可以起草纯技术 ADR，但草案状态只能为 `Draft`，不得由 Agent 自行标记为 `Accepted`；
- 涉及产品定位、当前阶段、私人数据存储或上传承诺、重大对外承诺的 ADR，必须由产品负责人明确确认；
- 不得通过新建 ADR 绕过产品决策边界、人工审批操作或生产高风险操作限制；
- 未经明确要求，不自动暂存、提交、推送或合并 ADR。

## 任务完成交接

每项修改任务最终至少说明：

1. 做了什么；
2. 用户可感知的变化；
3. 修改了哪些文件；
4. 运行了哪些测试及结果；
5. 哪些测试没有运行；
6. 已知风险；
7. 未完成或暂缓事项；
8. 是否需要更新文档；
9. 是否需要产品负责人确认；
10. Git 状态和是否存在未跟踪文件；
11. 是否改变数据事实来源、存储位置或传输边界；
12. 是否影响持久化 schema、版本、迁移、导入导出或恢复；
13. 是否采用临时方案，其技术债和迁移触发条件是什么；
14. ADR 状态是 `Not Required`、`Proposed`、`Draft`、`Accepted` 还是 `Deprecated`；
15. 是否产生跨入口或跨模块重复业务逻辑；
16. UI 任务是否复用现有 Token 和组件，是否新增公共视觉规则；
17. 是否完成与修改范围对应的架构、代码和设计检查。

较大任务开始前参考 `docs/ai/task-template.md`，完成后参考 `docs/ai/handoff-template.md`。

## 人工审批操作

未经产品负责人确认不得执行：

- 生产数据库迁移；
- 删除、覆盖或修改真实用户数据；
- 生产发布；
- 自动合并主分支；
- 修改环境密钥；
- 改变数据存储和隐私承诺；
- 公开用户资产数据；
- 大规模架构迁移；
- 创建超出当前阶段范围的大功能。

## 最后检查

开始前：

- 读对文档了吗？
- 明确任务范围和非范围了吗？
- 检查 Git 状态了吗？
- 是否涉及敏感数据或高风险操作？

完成前：

- 是否只改了范围内内容？
- 是否运行了必要测试？
- 产品说明与代码行为是否一致？
- 是否留下可审查 diff？
- 是否完成交接说明？
