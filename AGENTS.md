# Repository Guidelines

本仓库用于开发一个资产管理网站，核心模块包括资产录入、持仓价值计算、资产变化归因、投资复盘笔记和社区。任何实现都应优先保证数据准确性、审计可追踪性、安全性和合规表达。

## 项目定位

- 面向个人或团队资产管理场景，帮助用户记录资产、查看组合价值、分析收益变化来源、沉淀投资复盘，并在社区中交流观点。
- 网站可以展示资产表现和分析结果，但不得包装为自动投顾、收益承诺或个性化投资建议。
- 所有金融计算必须可解释、可复算、可测试。涉及金额、数量、汇率、价格和收益率的逻辑必须避免浮点误差。

## 推荐技术方向

项目目前尚未选择框架、包管理器、数据库、格式化工具或测试框架。除非用户另行指定，后续选型优先考虑：

- TypeScript 作为主要应用语言。
- React/Next.js 或同等能力的现代 Web 框架。
- 关系型数据库存储账户、资产、交易、估值快照、复盘笔记和社区内容。
- Decimal/BigInt/定点数方案处理金额、数量、价格、费率和收益率。
- Playwright 用于关键用户路径的端到端测试。

引入任何工具后，必须同步更新 `README.md` 中的安装、启动、构建、测试和环境变量说明。

## 建议目录结构

如果采用 TypeScript Web 项目，优先按以下边界组织代码；实际框架不同也应保留相同的领域划分：

```text
src/
  app/                  # 页面路由、布局、服务端入口或框架约定入口
  components/           # 跨模块复用 UI 组件
  features/
    assets/             # 资产录入、账户、交易、资产分类、附件
    portfolio/          # 持仓、估值、币种换算、收益率展示
    attribution/        # 资产变化归因、现金流、价格变动、汇率、费用拆解
    notes/              # 投资复盘笔记、标签、关联资产或交易
    community/          # 帖子、评论、点赞、举报、审核状态
  domain/               # 领域模型、金融计算、校验规则、权限规则
  server/               # API、服务层、任务队列、外部数据源适配
  db/                   # schema、migration、seed、查询封装
  lib/                  # 通用工具、日期、金额、日志、配置
  styles/               # 全局样式、设计 token
tests/
  unit/                 # 纯函数、领域模型、金融计算
  integration/          # API、数据库、权限、外部数据适配
  e2e/                  # 浏览器关键路径
docs/
  decisions/            # 重要架构决策记录
  compliance/           # 合规说明、风控清单、数据处理说明
```

## 开发规范

- 保持模块边界清晰：页面层只负责交互和组合，业务规则放在 `domain/` 或对应 feature 的领域文件中。
- 金融计算不得散落在 UI 组件内。价格、持仓、收益、归因和汇率换算必须通过可测试的纯函数或服务完成。
- 金额、数量和收益率必须使用明确单位和精度。字段名应体现含义，例如 `amountMinor`、`quantityDecimal`、`currency`、`pricedAt`。
- 日期时间统一使用带时区的存储格式。估值、交易发生时间、价格时间戳和展示时区必须区分。
- 数据写入应有校验、权限检查和审计信息。重要变更记录 `createdBy`、`updatedBy`、时间戳和必要的变更原因。
- 外部行情、汇率或链上数据应通过适配层接入，保留来源、拉取时间和失败降级策略。
- 社区内容必须支持举报、删除、隐藏、审核状态和基本反滥用措施。
- UI 应偏工具型和高信息密度，适合长期反复使用；避免营销页式布局作为主体验。
- 不要在代码、测试或文档中提交真实密钥、真实账户、真实身份证明、真实交易记录或用户隐私数据。

## 视觉与交互原则

- 整个网站应尽量简约，但必须足够清晰。简约不等于信息不足；关键资产、收益、风险、时间范围和数据来源必须一眼可见。
- 优先使用干净的布局、明确的层级、克制的颜色和稳定的间距。避免装饰性强但不提升理解效率的视觉元素。
- 主体验应像资产管理工具，而不是营销落地页。默认首页应直接进入资产概览、录入、分析或复盘等核心工作流。
- 信息密集区域使用表格、分组、筛选、排序、标签、状态和可折叠详情来提高可读性，不用大面积卡片堆叠稀释信息。
- 颜色主要用于表达状态和语义，例如盈利、亏损、风险、待确认、数据缺失；不要依赖单一颜色传递重要含义。
- 每个页面都应有清晰主任务和少量高优先级操作。次要操作放入菜单、更多操作或详情区，避免按钮泛滥。
- 表单应减少不必要字段，但资产、交易、价格、费用、币种、时间和来源等影响计算准确性的字段必须明确。
- 图表应服务于解释问题：组合价值、收益变化、归因拆解、资产分布和时间趋势。不要添加无法辅助决策的装饰图表。
- 移动端优先保证查看概览、快速录入、阅读笔记和社区浏览；复杂归因和批量管理可以在桌面端提供更完整体验。
- 不需要为“简约清晰”额外安装 skill。现有前端开发规范和已安装的 `playwright` 足够用于后续 UI 验证；如果将来需要从 Figma 稿实现，再考虑安装 Figma 相关 skill。

## 核心模块要求

### 资产录入

- 支持资产分类、账户、币种、交易类型、数量、价格、费用、备注和附件等基础字段。
- 录入表单必须有客户端和服务端校验，错误信息应明确指出字段和修正方式。
- 导入功能应先预览、校验和去重，再允许入库。

### 持仓价值计算

- 持仓价值应明确区分成本、现价、市值、现金流、已实现收益、未实现收益和费用。
- 多币种资产必须记录原始币种和折算币种，汇率来源与时间戳必须可追踪。
- 计算结果应可由交易流水、价格快照和汇率快照复算。

### 资产变化归因

- 至少区分净投入/提现、价格变动、汇率变动、分红/利息、手续费、税费和手动调整。
- 归因结果应能解释组合总价值从起始日期到结束日期的变化。
- 对无法归因或数据缺失的部分必须显式标记，不得静默吞掉差异。

### 投资复盘笔记

- 笔记可关联资产、账户、交易、时间区间和标签。
- 支持草稿、发布或私有状态，默认保护用户隐私。
- 复盘内容可以记录观点和决策过程，但展示层应避免暗示平台提供投资建议。

### 社区

- 用户生成内容必须有权限控制、可见性控制、举报和审核流程。
- 社区观点必须与用户个人资产数据隔离，禁止默认公开任何资产、持仓或交易信息。
- 展示投资观点时应有风险提示，不允许收益保证、跟单暗示或误导性排名。

## 测试要求

- 金融计算必须有单元测试，覆盖正常路径、边界精度、负数、零值、多币种、费用、分红、拆合股或等价复杂场景。
- 归因逻辑必须有固定样例测试，确保各项贡献加总后能对齐组合价值变化。
- API 和数据库写入必须有集成测试，覆盖鉴权、越权访问、校验失败、重复提交和事务回滚。
- 端到端测试至少覆盖资产录入、查看持仓、生成归因、创建复盘笔记和社区发帖/举报的关键路径。
- 测试数据必须使用虚构账户和虚构交易，不得包含真实个人信息或真实资产明细。
- 修复金融计算、权限、安全或隐私相关 bug 时，必须先补充或更新能复现问题的测试。

## 金融合规和安全注意事项

- 页面、邮件、通知和社区文案不得承诺收益、暗示保本、引导高风险投资或构成个性化投资建议。
- 必要位置应展示风险提示，例如“数据仅供记录和分析，不构成投资建议”。
- 明确区分用户自行录入数据、第三方数据和平台计算结果；展示外部行情时注明来源和延迟。
- 用户资产、交易、笔记和身份相关数据属于敏感数据。默认最小权限访问，传输加密，敏感字段加密或脱敏展示。
- 登录、会话、权限、邀请、分享链接和导出功能必须按安全敏感功能处理。
- 社区模块需要内容治理：举报、审核、封禁、删除、申诉记录和审计日志应可扩展。
- 导出、删除账户、数据迁移和隐私请求应保留清晰流程，避免不可逆误操作。
- 如接入券商、交易所、支付、行情或 KYC 服务，必须单独记录数据处理范围、密钥管理方式、速率限制、错误处理和合规责任边界。

## 代码审查清单

- 金额和收益计算是否使用了安全的数值类型，并有测试覆盖。
- 权限检查是否在服务端执行，用户是否只能访问自己的私有数据。
- 外部数据是否记录来源、时间戳和失败状态。
- 是否新增或修改了合规敏感文案。
- 是否引入真实用户数据、真实密钥或不应提交的配置。
- 是否同步更新 `README.md`、环境变量示例、migration 或测试说明。
