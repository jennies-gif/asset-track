# 资产轨迹 MVP 产品审计

审计日期：2026-06-13  
审计范围：根目录原生 JS MVP（`index.html`、`src/app.js`、`src/styles/app.css`、`src/domain/*`、`tests/*`、`README.md`）。仓库内 `apps/web` 仍是 React/TypeScript 重构入口示例，不是当前主体验。

## 审计结论

当前产品已经超过“静态 Demo”的阶段：资产总览、资产录入、持仓列表、加仓/减仓/清仓、收益分析、归因、复盘笔记、导入导出、本地存储、风险提示和 Demo 数据都已具备。它具备给种子用户试用的基础，但离“上线可用 + 用户愿意信任 + 有付费感”的 MVP 还有一轮产品化打磨。

最大问题不是功能缺失，而是：主体验仍显得功能堆叠，首次录入路径和专业可信表达没有足够聚焦；首页没有把“长期记录资产变化、看清收益来源、沉淀投资决策”的价值在首屏讲透；上线所需的隐私、条款、风险、备份、反馈和变更说明页面还缺失。

## 当前产品优点

### P0 优点

- 已有清晰的四个主导航：总览、资产、分析、笔记，基本符合 MVP 信息架构方向。
- 默认加载 Demo 数据，新用户首次进入不会看到空白产品，有助于理解产品能力。
- 资产模块已支持资产录入、当前持仓、历史持仓、交易记录，并支持买入、加仓、减仓、清仓。
- 清仓和卖出后会引导写复盘笔记，复盘模块已经和资产/交易产生了初步关联。
- 金融计算使用 BigInt/定点数思路，避免直接浮点计算；已有单元测试覆盖金额、汇率、持仓、归因、已实现收益等关键逻辑。
- 首页和分析页已展示风险提示、本地数据提示、行情说明、数据更新时间、数据质量等可信度信息。
- 支持 JSON 全量导出、CSV 资产导出、JSON 导入校验，本地数据边界表达较明确。
- 移动端已有响应式 CSS，表格支持横向滚动，主要布局有移动端降级。

### P1 优点

- 支持快速识别常见资产代码/名称，并自动带出类型、市场、币种等字段。
- 分析页不只是图表，已经尝试给出“风险与收益结论”“数据核对”“资产贡献排行”。
- 复盘笔记有模板、标签、搜索、草稿、资产和交易关联能力，方向符合长期留存。
- 有行情同步、本地 API、行情缓存、数据源文档等工程化基础。
- README 已写明 MVP 边界、风险提示、本地部署、测试和验收路径。

## 当前最大问题

### P0

1. 首页首屏过于“分析工作台化”，没有足够明确地引导用户完成第一笔记录。
   - 当前首页直接展示指标、曲线、基准、配置等内容。Demo 数据下很丰富，但真实用户更需要“当前资产状态 + 下一步行动 + 数据是否可信”。
   - 首屏主操作不够集中。顶部有“新增记录”，资产页有“添加资产”，但首页没有强主 CTA。

2. 添加资产流程仍然偏重。
   - 代码中已有“快速录入”分组，但新建资产时仍能看到加仓/减仓/清仓相关结构，以及较多资产元数据字段。
   - 对首次用户而言，账户类型、市场、币种、数量、成本价等字段一次性出现，仍像表单堆砌。
   - 当前没有明确的“快速添加资产 / 完整编辑资产 / 记录交易”三段式入口。

3. 上线基础页面缺失。
   - 当前没有独立 Landing Page、Privacy Policy、Terms、Risk Disclaimer、Changelog、Feedback、Data Backup Guide。
   - 页内已有提示，但外部可信材料不足，影响公开上线和付费转化。

4. 导入流程与 README 描述存在落差。
   - README 写到“粘贴 JSON 备份并通过预览校验后导入”，当前 UI 实际是粘贴后直接导入校验，没有独立预览、差异确认和去重说明。
   - 对资产数据产品来说，导入覆盖风险需要更强确认。

5. React/TypeScript 重构入口与当前主体验容易造成定位混乱。
   - `apps/web` 是工程化重构示例，但当前上线主入口是根目录 `index.html`。
   - 后续上线与 README 需要继续明确“当前生产入口”，避免部署错入口。

## 上线前必须修的问题

### P0

- 首页重构为更聚焦的 dashboard：总资产、累计收益/收益率、最近变化、资产配置、主要持仓、最近交易、最近复盘、数据完整度和一个明确主操作。
- 空状态重做：真实无资产状态不能只是“暂无当前持仓”，应解释区域用途，并提供“添加第一笔资产”和“加载示例数据”。
- 资产录入流程分层：默认只展示快速添加资产所需字段；完整编辑和记录交易通过明确入口进入。
- 增加数据可信说明：价格缺失、手动价格、汇率来源/时间、数据更新时间、本地保存状态需要在关键位置稳定展示。
- 补齐上线基础页面：隐私、条款、风险、备份、更新日志、反馈入口。
- 导入前增加确认或预览说明，至少明确“会替换当前本地数据 / 建议先导出备份”。
- 完整跑通 `npm test`、`npm run typecheck`、`npm run build`，并记录结果。

## 可以上线后再做的问题

### P1

- 更完整的导入去重、字段映射、冲突解决。
- 更细的交易类型：分红、转入、转出、调整成本等目前在菜单中出现但为禁用，后续再补。
- 更完善的行情源状态、失败重试和价格版本历史。
- 更可靠的移动端表格替代表达，例如重点数据卡片 + 详情抽屉。
- React/TypeScript 版本迁移成为主入口。
- 更细的权限、用户账户、服务端同步和数据库持久化。

### P2

- 云端账户系统和多设备同步。
- 自动行情源增强。
- 高级分析、AI 投资复盘助手、订阅付费。
- 公开观点广场和社区治理后台。

## 影响付费感的问题

### P0

- 产品价值没有形成“长期使用闭环”的第一印象。当前功能很多，但用户不一定立刻理解为什么值得持续记录。
- 首页缺少“数据完整度 / 备份状态 / 最近复盘缺口”等长期管理信号。
- 基础页面缺失会削弱用户对隐私、安全、风险边界的信任。

### P1

- UI 有专业基础，但部分地方仍像内部工具：按钮多、提示条多、表单长、页面层级偏满。
- 分析页使用了 MWR/TWR/SR 等专业名词，但当前切换控件更像展示态，容易造成用户认为功能已完整可用。
- 没有 Pro 能力预留，用户看不到未来付费价值边界。
- 复盘与资产/交易已有连接，但还可以更突出“复盘沉淀是资产管理的一部分”，而不是单独笔记本。

## 具体修改建议

### 信息架构

#### P0

- 保留四个主导航：Overview / 总览、Assets / 资产、Insights / 分析、Journal / 复盘。
- 将“笔记”命名为“复盘”，更贴合产品定位。
- 资产页内部明确三层：快速添加资产、持仓管理、交易记录。
- 分析页承接收益、配置、归因、风险和数据可信，不再承担过多数据维护入口。

#### P1

- 把备份入口从资产页“备份”菜单扩展为全局数据状态入口，但不要喧宾夺主。
- 将“同步价格”改为更专业的“更新行情”或“同步估值”，并在结果中展示数据源和时间。

### 首页 / 总览页

#### P0

- 首屏改成 dashboard：左侧总资产与收益，右侧数据状态和下一步行动。
- 增加“最近交易”“最近复盘”“主要持仓”“待补全数据”区块。
- 无资产时展示专业空状态：会展示什么、为什么重要、主 CTA、示例数据入口。

#### P1

- 弱化基准对比在首页的占比，移到分析页或折叠到二级区域。
- 曲线无数据时不要留大面积空白，改为轻量说明 + 操作。

### 资产录入和交易流程

#### P0

- 新建资产默认只问：资产名称/代码、分类、账户、币种、数量、成本价、开仓日期。
- 当前价格默认可选，缺失时明确“暂按成本价估值，稍后可补价格”。
- 记录交易入口从持仓行进入，打开只和该交易相关的表单：买入、卖出、清仓。
- 新建资产时隐藏加仓/减仓/清仓选择，避免首次用户困惑。

#### P1

- 增加即时估算反馈：成本金额、当前市值、未实现收益、卖出预计已实现收益。
- 表单关闭前如已填写内容，提示是否放弃。
- 表单移动端使用单列、底部固定提交区或更明显保存按钮。

### 数据可信度

#### P0

- 关键指标旁展示更新时间和估值口径。
- 缺失价格展示“待补价格”，不要让用户误以为价格准确。
- 手动价格、同步价格、默认成本价估值要有标签区分。
- 汇率设置要说明影响的是展示折算，不修改原始资产数据。

#### P1

- 分析页加入“计算口径说明”折叠区：总资产、市值、成本、收益率、归因。
- 导入/同步/编辑后记录更清晰的更新时间。

### UI 质感

#### P0

- 统一页面标题、卡片 padding、表格密度、按钮层级、空状态样式。
- 减少同时可见的提示条，把本地数据、风险、行情说明合并成更克制的数据状态条。
- 保证首页、资产页、分析页、复盘页视觉密度一致。

#### P1

- 正收益/负收益颜色保持语义，但降低大面积色块。
- 深色模式再做一次对比度和表格边界检查。
- 移动端将宽表格的关键列提炼为摘要，横向滚动作为补充。

### 商业化边界

#### P1

- 先写 `PRICING_STRATEGY.md`，不接支付。
- UI 中可以对高级导出、多币种高级分析、AI 复盘总结、自动备份提醒预留 Pro 标识。
- 不要锁死基础资产记录、基础交易记录、基础复盘和 JSON 导出，否则用户无法形成数据沉淀。

## 功能保留 / 弱化 / 延后

### 应保留

- 本地优先的数据保存模式。
- Demo 数据。
- JSON 导入导出。
- 资产录入、持仓列表、交易记录、清仓复盘引导。
- BigInt/定点数金融计算与单元测试。
- 风险提示和非投资建议表达。

### 应弱化

- 首页的基准对比和复杂分析展示。
- 社区/观点广场，目前应保持隐藏或移到上线后路线图。
- 过多顶部提示和过多同级按钮。
- 新建资产时展示交易调整表单。

### 应延后

- 云端账户、多设备同步、订阅支付。
- 完整社区。
- 高级行情源、券商/交易所同步。
- AI 投资复盘助手。

## 分阶段改造计划

### Phase 1：上线阻断问题修复

- 处理首页空状态、导入覆盖风险提示、关键数据可信标签。
- 补齐基础法律/风险/备份/反馈页面的轻量版本。
- 验证测试、类型检查和构建。

### Phase 2：首页和信息架构优化

- 导航命名优化为总览、资产、分析、复盘。
- 首页重构为核心 dashboard。
- 首页增加主要持仓、最近交易、最近复盘、数据完整度。

### Phase 3：资产录入和交易流程优化

- 新建资产改成快速添加。
- 完整编辑资产保留高级字段。
- 记录交易使用独立交易表单语义。
- 增加即时计算反馈和误关闭保护。

### Phase 4：分析、复盘、空状态优化

- 分析页增加可读结论和计算口径。
- 复盘页加强资产/交易关联。
- 全站空状态、错误状态、加载状态统一。

### Phase 5：商业化和上线材料补齐

- 输出 `PRICING_STRATEGY.md`。
- 预留轻量 Pro 标识。
- 补齐 Landing Page、隐私政策、使用条款、风险提示、更新日志、反馈入口和备份说明。

## 主要风险

- 改动首页和资产表单时容易破坏已有本地数据结构。应优先只改 UI 呈现和兼容逻辑，不迁移字段。
- 当前计算逻辑中部分趋势曲线使用模拟趋势推导，不应在 UI 中表达为真实历史收益。
- 行情同步依赖本地 API，线上静态部署时应明确“当前不会自动同步云端行情”。
- 社区能力虽然有代码雏形，但合规和治理不足，不适合当前 MVP 主推。
