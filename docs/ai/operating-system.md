# AI 协作运行机制

- 状态：当前协作机制基线
- 适用阶段：个人产品 MVP、上线准备与早期用户验证
- 最后更新时间：2026-07-11
- 维护责任：产品负责人确认，Codex 执行写入

## 一、协作机制目标

本文件定义个人产品从 MVP 构建、上线准备到早期用户验证阶段的固定协作机制。它不定义产品定位或具体需求，而用于说明产品如何按周期复盘、不同 AI 工具和 Skill 如何协作、产品结论如何转化为开发任务、文档如何持续更新，以及何时从单 Skill 升级到多个专业 Agent。

协作机制的目标是：

1. 避免产品开发只围绕局部功能推进，缺少全局复盘。
2. 让产品、用户体验、UI、技术和验证风险能够被周期性检查。
3. 保证每个周期只聚焦一个主目标，并控制个人开发者的任务数量。
4. 让产品决策、阶段状态和周度结论沉淀在仓库中，而不是只留在聊天记录。
5. 让 ChatGPT、Codex、GitHub 和未来的专业 Agent 使用同一套产品上下文。
6. 在流程稳定后，逐步将重复的信息收集和评审工作交给多个专业 Agent。

## 二、当前协作阶段

当前阶段：`单 Skill 周度评审机制建设阶段`

当前目标：

- 完成个人产品周度共创 Skill；
- 使用真实项目运行 3–5 次；
- 验证不同专业角色的评审边界；
- 验证固定输入和输出模板；
- 暂不自动编排多个 Agent；
- 暂不允许多个 Agent 自动修改代码。

当前项目状态以 [产品定位](../product/positioning.md) 和 [当前阶段](../product/current-stage.md) 为准，本文件不重复完整产品内容。

## 三、使用的软件与职责

### 产品负责人

由用户本人承担。

职责：

- 决定产品定位；
- 确认目标用户和核心价值；
- 对优先级和范围作最终决策；
- 审批重大产品、数据和技术决策；
- 决定是否进入下一阶段；
- 审批周度计划和 Codex 开发任务；
- 审查重要文档与代码 diff。

### ChatGPT

职责：

- 运行个人产品周度共创 Skill；
- 主持多角色产品复盘；
- 主动检查产品目标、核心流程和阶段风险；
- 挑战未经验证的假设；
- 协助形成产品解决方向；
- 在需要时运行产品解决方案 Skill；
- 在已有方案时运行产品方案评估 Skill；
- 生成周会结论和产品文档草稿；
- 给产品负责人的判断能力提供反馈。

限制：

- 不直接修改本地代码仓库；
- 不自行决定重大产品方向；
- 不把所有优化机会都转化为任务。

### VS Code

职责：

- 作为本地项目和文档的主要查看、编辑环境；
- 用于检查 Codex 修改后的文件和 Git diff；
- 不作为长期产品决策的独立存储位置，最终内容仍应进入 Git 仓库。

### Codex

职责：

- 读取仓库、代码和产品文档；
- 根据已确认结论创建或更新 Markdown 文档；
- 审查真实代码、测试和技术风险；
- 将产品任务转化为技术实施；
- 修改代码、运行测试并输出交接说明；
- 在任务完成后总结用户可感知变化、测试结果和遗留风险；
- 必要时整理 Git 变更摘要。

限制：

- 不得未经产品负责人确认修改产品定位、MVP 核心范围和重大决策；
- 不得未经批准扩大任务范围；
- 不得自行合并主分支、执行生产数据库迁移或发布高风险生产变更；
- 不得把未提交代码自动视为稳定产品能力。

### GitHub

职责：

- 保存代码、文档和决策历史；
- 提供 commit、branch、PR 和未来 Issue 管理；
- 作为正式版本历史，而不是产品判断来源本身。

### Supabase、Render、Vercel

当前职责：

- Supabase：数据库和后续用户数据能力；
- Render：API 和服务端能力；
- Vercel：前端部署。

当前阶段不要求它们自动参与周度评审。未来进入多 Agent 或小规模用户验证阶段后，再逐步接入：

- 部署状态；
- API 健康状态；
- 数据库迁移；
- 错误和运行异常。

## 四、周度运行触发规则

### 固定触发

每周运行一次产品周度复盘，建议默认在每周结束或下一周期开发前运行。

### 阶段触发

以下情况即使未到固定时间，也应运行一次阶段评审：

- 完成一个主要产品模块；
- 准备开始一批新的 Codex 开发任务；
- 产品定位或目标用户发生变化；
- 准备邀请种子用户；
- 准备冻结测试版本；
- 用户测试结束；
- 出现重大数据、隐私、信任或架构风险；
- 连续开发多个任务但未进行全局复盘。

### 轻量复盘

如果一个周期没有有效产品进展，仍应做轻量复盘，至少确认：

- 当前主目标是否仍然有效；
- 没有进展的原因；
- 是否需要缩小任务范围；
- 下一周期是否继续同一目标。

## 五、周会前材料

### 最小输入

每次周会至少准备：

- 上周期唯一主目标；
- 上周期关键任务；
- 实际完成情况；
- 未完成事项及原因；
- 当前可投入资源；
- 当前最担心的问题。

### 标准输入

尽量增加：

- `docs/product/positioning.md`；
- `docs/product/current-stage.md`；
- 上一期周会记录；
- 当前页面截图或部署地址；
- 本周期 GitHub 变更摘要；
- 已知 Bug；
- UI 问题；
- Codex 技术审查摘要；
- 用户反馈或测试结果；
- 产品负责人对当前问题的判断。

### 上线后输入

未来进入真实用户阶段后，再增加：

- 用户完成情况；
- 首次录入时间；
- 关键流程流失；
- 用户反馈；
- 部署异常；
- API 或数据库错误；
- 继续使用意愿；
- 核心假设验证结果。

信息不足时仍可分析，但所有推断必须标记为假设或低置信度结论。

## 六、个人产品周度共创 Skill 固定议程

该 Skill 以一个完整产品阶段或复盘周期为最小工作单元，不用于日常按钮、文案和低影响细节决策。

### 1. 上周期复盘

检查：

- 上周期目标是否合理；
- 实际完成了什么；
- 哪些工作真正推进了核心目标；
- 哪些工作只是增加代码或功能，但没有推进价值验证；
- 未完成原因；
- 是否存在范围膨胀、返工或偏航。

### 2. 当前阶段判断

重新判断：

- 当前产品阶段；
- 是否满足进入下一阶段的条件；
- 是否看似做了很多，但核心闭环仍未成立；
- 是否需要暂停功能开发，先处理定位、信任或技术风险。

### 3. 多角色评审

#### 产品负责人视角

检查：

- 定位是否仍然聚焦；
- 当前工作是否服务第一价值；
- 核心闭环是否完整；
- MVP 范围是否膨胀；
- 当前主要矛盾是什么。

#### 产品与用户体验视角

检查：

- 用户是否理解产品价值；
- 主流程是否自然；
- 首次体验是否存在阻断；
- 信息架构和操作顺序是否合理；
- 用户是否知道下一步做什么。

#### UI 设计视角

检查：

- 视觉层级；
- 主次关系；
- 布局、间距和对齐；
- 组件一致性；
- 信息密度；
- 状态表达；
- 移动端；
- UI 是否影响产品可信感。

UI 角色应优先指出最影响理解、任务完成和整体品质的问题，而不是罗列所有可美化细节。

#### 技术负责人视角

当前单 Skill 阶段主要基于：

- Codex 技术审查摘要；
- Git 变更；
- 测试结果；
- 已知技术债；
- 数据和 API 变化。

重点检查：

- 数据正确性；
- 隐私和安全；
- 主流程稳定性；
- 技术架构是否阻碍当前验证；
- 是否存在过度重构；
- 是否增加个人开发者不可承受的维护成本。

真实全仓代码审查由 Codex 执行，不由产品 Skill 假装完成。

#### 用户研究与验证视角

检查：

- 哪些是已确认事实；
- 哪些是产品负责人判断；
- 哪些只是 AI 推断；
- 哪些是待验证假设；
- 当前测试是否能够区分产品价值问题、流程问题和技术问题；
- 下一阶段最需要收集什么证据。

#### 风险与质量视角

检查：

- 隐私；
- 数据丢失；
- 计算错误；
- 备份与恢复；
- 行情缺失；
- 对外承诺；
- 合规边界；
- 部署和回滚。

### 4. 主要矛盾收敛

各角色可以分别指出问题，但最终由统一的产品负责人视角收敛为：

- 当前一个主要矛盾；
- 最多三个关键问题；
- 一个下周期主目标；
- 最多四个关键任务。

不得将每个角色的完整问题列表直接转化为开发待办。

### 5. 下周期计划

必须输出：

- 下周期唯一主目标；
- 2–4 个关键任务；
- 每项任务的目的；
- 完成标准；
- 是否需要调用其他 Skill；
- 明确不做事项；
- 下次复盘触发条件。

## 七、Skill 调用规则

### 个人产品周度共创 Skill

适用：

- 每周产品复盘；
- 阶段判断；
- 全局问题扫描；
- 下周期计划；
- 上线准备；
- 用户测试复盘；
- 判断是否偏离 MVP。

核心问题：`当前阶段最值得解决什么，以及下一周期应该做什么。`

### 产品解决方案 Skill

适用：

- 已经确认问题；
- 还没有明确解决方案；
- 需要比较不同产品方案；
- 需要控制 MVP 实现范围。

核心问题：`这个明确问题可以怎么解决。`

### 产品方案评估 Skill

适用：

- 已有明确产品方案；
- 需要评估用户流程、交互、边界和风险；
- 需要判断是否可以进入开发；
- 需要检查方案是否过度设计。

核心问题：`这个方案是否足够合理、完整且适合当前阶段。`

### 直接交给 Codex

以下情况通常不需要完整产品 Skill：

- 明确技术 Bug；
- 已有清晰验收标准的小型修复；
- 纯技术任务；
- 不影响产品逻辑的低风险实现；
- 已完成方案评审的开发任务。

### 调用衔接

周度共创 Skill 输出时，应标记：

- 继续在周会中澄清；
- 进入产品解决方案 Skill；
- 进入产品方案评估 Skill；
- 直接生成 Codex 任务；
- 需要 Codex 先做技术审查。

## 八、周会输出限制

每次完整周会应遵守：

- 只能有一个下周期主目标；
- 最多三个关键问题；
- 最多四个关键任务；
- 每项任务必须有完成标准；
- 必须有“明确不做”；
- 必须标记事实、推断和假设；
- 必须说明下一步应调用的 Skill 或执行工具；
- 不得将所有 UI、技术和产品问题同时纳入下周期；
- 不得因为某项能力长期需要，就提前把它列为当前门槛。

## 九、周会后写入规则

每次周会结论经产品负责人确认后，按需更新以下内容。

### 必须更新

- `../product/weekly-reviews/YYYY-Www.md`；
- `../product/current-stage.md`。

### 视情况更新

- `../product/roadmap.md`；
- `../product/changelog.md`；
- `../decisions/`；
- `context.md`。

### 高风险文档

以下文件只能在产品负责人明确批准后修改：

- `../product/positioning.md`；
- MVP 核心范围相关文档；
- 重大产品决策记录；
- 数据存储和隐私承诺；
- 阶段切换结论。

### 写入流程

1. ChatGPT 或 Skill 形成结论；
2. 产品负责人确认；
3. Codex 按明确指令写入；
4. Codex 展示 diff；
5. 产品负责人检查；
6. 再决定是否暂存和提交 Git。

## 十、Codex 开发执行流程

当周会确认某项任务进入开发时：

1. 使用 `task-template.md` 生成任务说明；
2. 明确任务目标；
3. 明确本次范围；
4. 明确不在范围；
5. 列出涉及页面、流程、数据和 API；
6. 定义边界状态；
7. 定义验收标准；
8. 要求 Codex 先检查现状并输出实施计划；
9. 产品负责人确认后再修改代码；
10. 运行必要测试；
11. 使用 `handoff-template.md` 进行交接；
12. 更新产品 Changelog 或相关文档草稿；
13. 产品负责人进行产品验收；
14. 必要时调用产品方案评估 Skill 做实现复核。

## 十一、当前多 Agent 升级路线

### 阶段一：单 Skill 周度评审

当前阶段。

特点：

- ChatGPT 运行周度共创 Skill；
- Codex 提供真实代码和技术检查；
- 产品负责人手动组织材料；
- 每次评审后手动让 Codex 写入文档。

### 阶段二：人工编排专业 Reviewer

周度 Skill 稳定运行 3–5 次后，考虑在 Codex App 中建立：

- Product Lead；
- Product & UX Reviewer；
- UI Design Reviewer；
- Engineering Reviewer。

初期全部为只读 Reviewer，不自动修改代码。

### 阶段三：增加 QA 与发布角色

接近种子测试或公开上线时增加 QA & Release Reviewer。

职责：

- 主流程回归；
- 移动端；
- 错误状态；
- 部署检查；
- 回滚；
- 发布门禁。

### 阶段四：半自动负责人 Agent

当手动收集和组织成本明显升高后，再考虑：

- 自动读取 GitHub 变更；
- 自动汇总部署状态；
- 自动调用专业 Reviewer；
- 自动生成候选周计划；
- 经用户批准后创建 Issue 或 Codex 任务。

## 十二、升级到多个专业 Agent 的判断标准

以下条件出现约 4 项时，可以开始人工编排多个专业 Agent：

1. 周度共创 Skill 已稳定运行至少 3 次；
2. 各角色检查内容基本固定；
3. 产品、UI 和技术角色的结论具有互补性；
4. 单次评审涉及多个核心页面或较大代码范围；
5. 一个上下文难以同时深入产品、UI 和代码；
6. 每周重复整理 Git 变化、截图和技术材料；
7. 同一周期存在多个可以并行的独立评审任务；
8. 输入和输出模板已经稳定；
9. 已有统一项目上下文；
10. 产品负责人能明确说明哪些操作必须人工审批。

以下情况不应升级：

- 产品定位仍频繁变化；
- 每周只有一个明确的小问题；
- 角色结论高度重复；
- 项目文档没有持续维护；
- 还无法明确 Agent 权限；
- 尚未验证周度评审本身有价值。

## 十三、人工审批边界

任何 Agent 都不得未经确认执行：

- 修改产品定位；
- 扩大 MVP 核心范围；
- 改变目标用户；
- 修改隐私和数据存储承诺；
- 创建重大产品决策；
- 执行生产数据库迁移；
- 删除生产数据；
- 自动合并主分支；
- 自动发布高风险生产版本；
- 自动公开用户资产或复盘；
- 自动输出个性化投资建议。

可以在低风险条件下自动生成草稿或建议：

- Git 变更摘要；
- 测试结果；
- 技术检查报告；
- 产品 Changelog 草稿；
- 周会记录草稿；
- task 和 handoff 文档；
- 低风险 Bug 修复建议。

| QA Agent 引入 | 未开始 | 进入种子测试或正式上线准备 |
| 半自动负责人 Agent | 未开始 | 手动协调成本明显升高 |
