# 0001: 上线顺序和数据边界

日期：2026-04-29

## 状态

Accepted

## 背景

资产轨迹已经有一个无外部依赖的本地 MVP 原型，但要上线为 To C 产品，需要先稳定领域模型、金融计算、权限边界和行情 / 净值数据边界。否则后续 React + TypeScript、数据库、API 和数据源接入会围绕 UI 字段反复返工。

## 决策

按以下顺序推进：

1. 先完成 PRD、领域数据模型、数据库 schema 草案和 API 契约。
2. 再进行 React + TypeScript 工程化重构。
3. 金融计算和归因逻辑必须保持在领域层，并持续用单元测试覆盖。
4. 数据库和 API 落地后，所有私有数据必须服务端鉴权，不能依赖客户端过滤。
5. 行情 / 净值数据首版只覆盖沪深 300、纳斯达克 100、恒生科技、ETF 白名单和国内公募基金净值。
6. 社区、支付订阅和复杂商业化后置。

## 数据边界

- 股票和 ETF 使用日收盘价。
- 场外基金使用单位净值，允许保存累计净值和复权单位净值。
- 港股首版只覆盖恒生科技指数成分股，不覆盖港股全市场。
- 所有价格、净值和汇率都必须保存来源、日期、拉取时间和质量状态。
- 数据缺失必须显式展示，允许用户继续使用手动录入价格。

## 数值边界

- 金额使用最小单位整数。
- 数量、价格、基金净值和汇率使用定点十进制。
- 收益率使用 basis points。
- 不使用 JavaScript `number` 或数据库浮点类型作为金融计算事实来源。

## 合规边界

- 产品不提供投资建议。
- 页面和社区不得承诺收益、暗示保本或引导跟单。
- 社区观点必须与用户私有资产、持仓、交易和笔记隔离。
- 外部行情和基金净值展示时必须注明来源和延迟。

## 后果

正面：

- 后续重构和后端开发有稳定合同。
- 金融计算、权限和数据来源更容易测试和审计。
- 首版数据范围可控，避免过早做复杂行情系统。

代价：

- 上线前需要维护更多文档和 schema。
- 原型阶段不会立即拥有完整真实行情和云同步能力。
- React + TypeScript 重构必须遵守这些数据边界，不能只按页面组件拆分。
