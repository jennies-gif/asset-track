# 市场资源库和行情抓取边界

日期：2026-07-11

## 背景

资产资源库会随着 A 股、港股、美股、基金、ETF、数字资产和贵金属覆盖扩大而持续增长。它来自外部公开数据源，需要记录来源、更新时间、别名、覆盖率和同步状态。

此前将完整资源库生成进前端源码，会导致前端包变重、Git diff 巨大、资源库更新像代码变更，并且容易把“可搜索资产主库”误用为“行情抓取列表”。

## 决策

完整资产资源库是服务端主数据，不是前端代码资产，也不是默认行情抓取清单。

- 前端只保留轻量 seed / fallback，不内置完整资源库。
- 完整资源库写入 PostgreSQL，由 API 查询。
- 本地 `storage/market-data/instrument-registry.json` 只作为导入缓存和无数据库时的开发回退。
- 行情抓取不得默认扫完整资源库。

## 行情触发规则

价格、净值和汇率抓取只允许由以下触发器启动：

1. 用户搜索或 lookup 某个代码时，只按需抓取该代码的最新价格或净值。
2. 用户首次录入资产后，按该资产的首次持有日期到今天创建一次历史回补任务。
3. 所有用户录入过且有代码的资产，每天定时抓取最新价格或净值。
4. 少量系统默认基准，每天定时抓取。
5. 用户点击“同步价格”时，只实时抓取该用户录入过的资产。

如需临时尝试完整资源库行情抓取，必须显式传入 `--all-registry=true`，不得作为默认行为。

## 历史回补任务

历史回补任务以用户资产为粒度，不以完整资源库为粒度。

- 任务表：`market_data_backfill_tasks`。
- 幂等键：`user_id + asset_id + symbol + market + date_from + date_to`。
- 初始状态：`pending`。
- 允许状态：`pending`、`running`、`completed`、`partial`、`failed`、`cancelled`。
- 触发来源默认是 `asset_created`；后续可以扩展为导入确认、修正持有日期、手动重跑。
- API 只负责创建任务；真正抓取由后续 worker 或计划任务消费，避免资产录入请求被长时间阻塞。

## 约束

- 可增长、来自外部、需要同步或审计的数据不得打进前端 bundle。
- 大型自动生成文件进入源码目录应视为架构警报。
- 资源库同步脚本必须保留覆盖率摘要和来源记录。
- 行情脚本必须按用户资产、指定 symbols、指定 universe 或系统基准收敛范围。
- 多年历史行情不得对全资源库无差别抓取，应按用户持仓、关注列表、基准或明确任务按需缓存。

## 验证

- 单元测试覆盖默认行情选择不会使用完整资源库。
- `npm test`
- `node --check scripts/market-data/fetch-market-data.mjs`
