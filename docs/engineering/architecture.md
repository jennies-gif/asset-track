# Asset Trail Engineering Architecture

## 1. 文档目的

本文档定义 Asset Trail 项目的长期工程架构原则。

目标是在快速 MVP 迭代与长期可维护性之间取得平衡，并指导：

- AI 辅助开发；
- 新功能设计；
- 数据模型设计；
- 模块拆分；
- 技术决策；
- 架构演进。

核心原则：

> 新需求既要服务当前阶段目标，也要考虑长期架构正确性，避免为了局部快速实现而持续扩大不可迁移的技术债。

本文档描述工程演进方向，不代表所有长期架构能力都必须在当前 MVP 阶段完成。

当前 MVP 可以存在简化版本，不要求所有模块立即满足完整分层。

---

## 2. 当前架构定位

当前处于种子用户试用版本准备阶段，工程架构正在从快速验证实现逐步向可持续迭代方向演进。

当前项目同时存在：

- 原生 JavaScript MVP 主入口；
- React + TypeScript 并行开发入口；
- Node API 服务；
- PostgreSQL 数据模型；
- `localStorage` 本地优先数据模式；
- 公共行情及相关服务端能力。

当前主要数据路径包括：

```text
Frontend
    ↓
localStorage
```

以及：

```text
Frontend
    ↓
API Layer
    ↓
Database / External Data Source
```

当前原生 JavaScript 静态 MVP 是种子用户试用版本的主体验候选。React + TypeScript 是长期演进方向，不代表当前正在进行主入口强制替换。

当前阶段工程工作的优先级是：

1. 保证私人资产数据边界与产品说明一致；
2. 保证核心路径稳定、可信、可恢复；
3. 冻结并验收种子用户试用版本；
4. 在不影响当前验证目标的前提下改善可维护性；
5. 避免新增不可迁移的前端硬编码和重复业务逻辑。

---

## 3. 长期架构目标

长期目标是逐步形成以下分层架构：

```text
Frontend
    ↓
Business Service Layer
    ↓
API Layer
    ↓
Database
```

各层长期职责如下：

- Frontend：负责展示、交互和用户体验；
- Business Service Layer：负责业务流程编排和业务规则；
- API Layer：负责服务入口、权限、校验和数据访问边界；
- Database：负责需要云端持久化的业务数据和一致性管理。

长期演进方向包括：

- 数据模型统一；
- 业务逻辑集中管理；
- 金融计算可解释、可复算、可测试；
- 支持多用户和多设备；
- 支持安全的云端同步、备份和恢复；
- 保持私人资产数据存储与传输边界透明；
- 降低不同前端入口之间的逻辑重复。

以上是长期目标，不构成当前种子用户试用版本的全部上线门槛。

## 4. Source of Truth（数据事实来源原则）

### 4.1 核心原则

每类业务数据必须拥有明确的数据事实来源。

设计和实现数据能力时，必须说明：

- 数据由谁产生；
- 数据保存在哪里；
- 哪一份数据是当前事实来源；
- 数据是否会发送到服务器；
- 数据如何读取、更新、导出、删除和恢复；
- 数据未来是否需要同步；
- 数据出现冲突时如何处理。

禁止同一个业务实体长期存在多个互相独立、无法确定优先级的数据来源。

### 4.2 当前 MVP 过渡策略

当前 MVP 允许 `localStorage` 作为私人资产数据的 local-first storage 方案。

可能包括：

- 用户账户；
- 资产与持仓；
- 交易和现金流；
- 用户设置；
- 私人复盘内容。

使用 `localStorage` 时必须：

- 明确其保存的数据范围；
- 明确数据不会被误认为已经云同步；
- 对解析失败和数据损坏提供明确错误提示；
- 禁止静默回退为正常 Demo 数据；
- 提供与当前阶段相匹配的导出、导入和恢复能力；
- 评估数据结构未来是否可以迁移；
- 避免无结构、无版本、无校验的数据持续增长。

### 4.3 新增数据规则

新增数据不要求在当前阶段全部立即通过 Database 和 API 保存，但必须在设计前明确：

1. 数据归属；
2. 当前事实来源；
3. 当前存储方式；
4. 是否包含私人或敏感信息；
5. 是否会发送到服务器；
6. 长期迁移方向；
7. 是否需要多设备同步；
8. 当前实现会形成什么技术债；
9. 未来迁移是否需要数据版本、转换或兼容策略。

禁止在没有数据归属和迁移判断的情况下，将新增业务数据直接实现为：

- 无结构的 `localStorage` JSON；
- 页面内部长期状态；
- 无法追踪来源的前端常量；
- 与既有模型重复的数据副本；
- 无法导出、校验或迁移的私有格式。

## 5. Data Ownership（数据归属原则）

所有数据设计前必须先判断数据类型和归属。

### 5.1 Static Configuration（静态配置）

特点：

- 所有用户基本一致；
- 不由用户产生；
- 通常不需要长期保存；
- 不涉及用户私人资产数据。

例如：

- 货币列表；
- 展示枚举；
- UI 配置；
- 固定状态选项。

可以存储于：

- 前端常量；
- 类型定义；
- 配置文件。

如果配置需要服务端动态更新、灰度发布或按用户区分，则应重新评估其归属。

### 5.2 User Business Data（用户业务数据）

特点：

- 由用户产生或修改；
- 需要长期保存；
- 可能包含敏感资产信息；
- 未来可能需要备份、恢复或同步。

例如：

- 用户账户；
- 投资资产；
- 持仓；
- 交易记录；
- 现金流；
- 收益记录；
- 私人投资复盘。

当前 MVP 可以采用 local-first storage。

长期方向可以根据产品验证结果演进为：

```text
Frontend
    ↓
Business Service / Sync Layer
    ↓
API
    ↓
Database
```

任何云端化或同步能力都必须先明确：

- 用户知情与控制；
- 上传的数据范围；
- 身份认证和权限；
- 加密与敏感数据保护；
- 同步冲突处理；
- 导出和删除；
- 失败与回滚策略。

### 5.3 Market Data（市场数据）

特点：

- 来自外部数据源；
- 会定期更新；
- 与用户资产估值相关；
- 不等同于用户私人资产数据。

例如：

- 股票价格；
- 基金净值；
- 汇率；
- 证券基础信息。

推荐长期结构：

```text
Market Data Source
    ↓
Market Data Service
    ↓
Cache / Database
    ↓
Asset Valuation
```

市场数据必须尽量保留：

- 数据来源；
- 数据时间；
- 拉取时间；
- 新鲜度状态；
- 缺失状态；
- 失败状态；
- 手动或估算标识。

公共行情云端获取不代表私人资产数据已经云同步。

## 6. Domain Model Principles（领域模型原则）

核心原则：

> 业务领域模型优先于页面数据结构。

不要按照以下方式设计：

```text
页面需要什么字段
    ↓
临时创建页面专用业务结构
```

应该优先考虑：

```text
Business Domain Model
    ↓
Storage Model
    ↓
API Contract
    ↓
Frontend View Model
```

当前 local-first 模式下也应保持领域模型清晰：

```text
Business Domain Model
    ↓
Local Storage Schema
    ↓
Frontend View Model
```

领域模型需要明确：

- 实体身份；
- 字段语义；
- 单位和币种；
- 精度；
- 时间语义；
- 数据状态；
- 实体关系；
- 校验规则；
- 版本和迁移方式。

页面展示结构可以基于领域模型派生，但不应反向决定核心业务模型。

---

## 7. Asset Domain Principles（资产领域原则）

资产管理系统长期需要支持多种资产类型，例如：

- 股票；
- ETF；
- 基金；
- 加密资产；
- 现金；
- 其他投资品。

因此应避免将单一资产类型作为整个系统的唯一模型。

例如，不应把以下结构直接作为通用资产模型：

```text
Stock {
  symbol,
  shares
}
```

统一资产领域模型应能够表达：

- 资产标识；
- 资产类型；
- 名称；
- 市场；
- 币种；
- 数量；
- 价格或估值方式；
- 数据来源；
- 更新时间；
- 缺失、过期、手动或估算状态；
- 与账户、交易和持仓的关系。

不同资产类型的特殊字段可以通过明确的扩展模型表达，不应全部堆积在一个缺乏约束的通用对象中。

---

## 8. Transaction Model Principles（交易模型原则）

交易属于独立业务实体。

长期领域关系可以表达为：

```text
Account
    ↓
Transaction
    ↓
Position
    ↓
Valuation
```

资产、账户、交易、持仓和估值之间应建立明确关系，但不应假设所有资产都一定具有相同的交易结构。

禁止长期将所有交易记录仅作为嵌套数组保存在资产对象中。

原因是交易需要独立支持：

- 查询；
- 编辑；
- 校验；
- 分析；
- 统计；
- 历史追踪；
- 成本计算；
- 现金流计算；
- 数据迁移。

当前 MVP 若使用简化或嵌套结构，必须明确其限制和长期迁移方向。

---

## 9. Frontend Architecture

### 9.1 Frontend 职责

Frontend 负责：

- 页面展示；
- 用户交互；
- 前端状态管理；
- API 调用；
- local-first 数据能力的界面编排；
- 加载、空白、错误和恢复状态；
- 用户体验。

### 9.2 Frontend 不应长期承担

Frontend 不应长期独立承担：

- 核心业务规则；
- 分散的金融计算；
- 无边界的数据持久化；
- 云同步规则；
- 权限判断；
- 数据库访问；
- 难以测试的复杂数据转换。

当前 local-first storage 可能由前端侧触发，但数据读写、校验、版本迁移和错误处理应通过明确的服务边界管理，不应散落在页面组件中。

---

## 10. Business Logic Layer

业务逻辑应该集中管理，并与具体页面解耦。

例如，资产收益计算不应直接定义在组件中：

```text
Component
    ↓
calculateReturn()
```

应该通过领域层或服务层提供：

```text
Domain / Service
    ↓
calculateReturn()
```

适合放在领域层的能力包括：

- 金额和数量计算；
- 持仓计算；
- 收益计算；
- 汇率换算；
- 估值；
- 数据状态判断；
- 业务校验。

适合放在服务层的能力包括：

- API 请求；
- local storage 访问；
- 数据导入和导出；
- 数据同步；
- 市场数据获取；
- 外部系统适配。

原则：

> 页面应该组合业务能力，而不是定义核心业务规则。

## 11. Module Organization

长期推荐按照职责组织模块：

```text
src/
  features/
    assets/
    accounts/
    transactions/
    analysis/

  domain/
  services/
  ui/
  state/
  utils/
```

该结构是演进参考，不要求当前阶段一次性重构现有目录。

### 11.1 features

负责具体业务功能，包括：

- 页面逻辑；
- 用户流程；
- 功能组件；
- 与特定业务场景相关的状态组合。

### 11.2 domain

负责：

- 纯业务模型；
- 金融计算；
- 业务校验；
- 数据状态规则。

`domain` 不应直接访问：

- DOM；
- API；
- `localStorage`；
- 数据库；
- UI 组件。

### 11.3 services

负责外部能力和副作用，例如：

- API 请求；
- local storage；
- 数据同步；
- 数据导入和导出；
- 市场数据；
- 外部服务适配。

### 11.4 ui

负责通用视觉组件，例如：

- charts；
- badges；
- empty states；
- status indicators；
- dialogs。

通用 UI 组件不应包含特定资产业务规则。

### 11.5 state

负责前端状态组织和跨组件状态协调。

持久化数据和临时界面状态应明确区分，避免把前端状态容器误当作业务数据事实来源。

### 11.6 utils

只存放通用、无业务语义的辅助能力。

具有明确业务含义的逻辑应进入 `domain` 或对应 `feature`，不能为了方便统一放入 `utils`。

## 12. React / TypeScript Evolution Strategy

当前项目同时存在原生 JavaScript MVP 与 React + TypeScript 入口。

React + TypeScript 是长期工程演进方向，但不作为当前种子用户试用准备阶段的强制迁移要求。

当前阶段不得为了技术栈统一而进行大规模重写，也不得默认将 React 入口视为正式种子用户主入口。

建议的渐进式路径：

```text
Legacy MVP
    ↓
JavaScript 模块边界清晰化
    ↓
可测试的 Domain Model
    ↓
按需引入 TypeScript
    ↓
逐步形成 React Application
```

新功能的技术选择需要结合：

- 是否影响当前冻结版本；
- 是否需要复用领域模型；
- 是否会扩大新旧入口之间的重复逻辑；
- 是否能够被后续迁移；
- 是否引入当前阶段不必要的维护成本。

适合优先采用 TypeScript 的场景包括：

- 新增复杂领域模型；
- 金融计算；
- API Contract；
- 跨模块共享类型；
- 数据校验与迁移。

旧功能应按风险和价值逐步迁移，不为了追求技术统一而进行一次性重写。

---

## 13. API Architecture Principles

API 负责：

- 服务端业务入口；
- 请求校验；
- 数据访问；
- 权限控制；
- 公共数据服务；
- 未来同步能力。

当前 local-first 私人资产数据可以不经过 API。

如果某类私人数据需要发送到服务端，必须先明确：

- 发送目的；
- 数据范围；
- 用户知情；
- 权限和身份；
- 保存时间；
- 删除和导出；
- 失败处理。

API 内部长期应逐步形成：

```text
Routes
    ↓
Services
    ↓
Repositories
    ↓
Database / External Data Source
```

应避免由单个文件同时承担：

- 路由；
- 业务规则；
- 数据访问；
- 外部服务适配；
- 错误处理。

API 变更需要同步检查调用方、API Contract、校验失败、错误状态和兼容性。

---

## 14. Database Principles

数据库是长期架构中需要云端管理的数据事实来源，但不是当前所有私人资产数据必须立即迁移的目标。

数据库设计应遵循：

- 数据模型明确；
- 字段语义清晰；
- 避免无必要的重复存储；
- 保持数据一致性；
- 支持未来扩展；
- 支持迁移和回滚；
- 考虑已有数据兼容；
- 保留必要审计信息。

涉及私人资产数据时，还必须考虑：

- 用户隔离；
- 最小权限；
- 身份认证；
- 敏感数据保护；
- 数据导出；
- 数据删除；
- 备份和恢复；
- 同步冲突。

数据库 schema 或核心数据关系发生重要变化时，需要记录 Architecture Decision。

---

## 15. Migration Principles

MVP 阶段允许使用：

- `localStorage`；
- 临时 mock；
- 简化数据结构；
- 阶段性兼容代码。

但临时实现必须满足：

- 边界清楚；
- 用户不会误解数据状态；
- 不损害数据正确性；
- 不静默吞掉错误；
- 可以测试；
- 可以识别和迁移；
- 不持续扩大不可迁移的前端硬编码。

每次新增数据或功能时，需要判断：

- 是否应该进入 Domain Model；
- 当前应保存在本地还是服务端；
- 是否需要 API；
- 是否需要数据版本；
- 是否会与现有来源重复；
- 长期如何迁移；
- 当前技术债是否可接受；
- 迁移触发条件是什么。

迁移工作应按真实产品需求和验证结果启动，而不是仅因为长期架构中存在更完整的技术方案。

---

## 16. Cloud Sync Evolution Principles

未来云同步需要建立在清晰的数据边界和已验证的用户需求之上。

演进前必须明确：

- 哪些数据需要同步；
- 哪些数据继续只保存在本地；
- 用户如何选择和控制；
- 首次上传如何确认；
- 本地与云端谁是事实来源；
- 离线修改如何处理；
- 冲突如何发现和解决；
- 删除如何同步；
- 失败如何恢复；
- 如何避免数据丢失或重复。

可能的长期结构：

```text
Local Data Store
    ↓
Sync Service
    ↓
API
    ↓
Cloud Database
```

云同步不应通过页面组件直接实现，也不应将公共行情请求与私人资产上传混为一谈。

在产品负责人确认数据和隐私边界前，不得擅自扩大私人资产数据的服务器传输范围。

---

## 17. AI Development Rules

AI 修改代码前必须判断：

1. 数据应该属于哪一类？
2. 当前事实来源是什么？
3. 数据应该保存在本地还是服务端？
4. 是否涉及私人或敏感数据？
5. 是否会改变真实的数据传输边界？
6. 业务逻辑应该属于哪个层？
7. 是否违反已有领域模型或 API Contract？
8. 是否产生重复逻辑？
9. 是否扩大不可迁移的前端硬编码？
10. 当前实现的长期迁移方向是什么？
11. 技术债是否影响当前验证、正确性或维护？
12. 是否需要记录 Architecture Decision？

AI 不得自行决定：

- 改变私人数据存储和上传承诺；
- 将完整云同步提升为当前 MVP 门槛；
- 启动大规模 React/TypeScript 主入口替换；
- 改变产品定位或当前阶段；
- 执行生产数据库迁移；
- 修改真实用户数据。

发现更优架构方案时可以提出建议，但未经产品负责人确认不得扩大实施范围。

---

## 18. Architecture Decision Requirement

以下情况必须在 `docs/decisions/` 中记录 Architecture Decision：

- 核心数据模型变化；
- 数据事实来源变化；
- 私人数据存储或传输边界变化；
- local-first 与云端同步策略变化；
- API 核心边界变化；
- 数据库 schema 或关键实体关系变化；
- 核心金融计算口径变化；
- 主应用架构或技术栈迁移；
- 影响多个模块的核心技术取舍。

通常不需要记录：

- 普通 UI 修改；
- 不改变业务语义的小功能；
- 局部代码整理；
- 临时调试；
- 不影响架构边界的缺陷修复。

Decision Log 应至少说明：

- 背景；
- 决策；
- 备选方案；
- 选择原因；
- 数据和隐私影响；
- 兼容与迁移影响；
- 风险；
- 回滚方式。

---

## 19. Architecture Review Checklist

新增或修改功能前，应检查：

### 数据

- 数据归属是否明确？
- 当前事实来源是否唯一？
- 是否涉及私人数据？
- 是否会发送到服务器？
- 是否具有校验、版本和恢复策略？
- 长期迁移方向是否明确？

### 业务逻辑

- 是否属于领域层或服务层？
- 是否把核心金融逻辑写入了 UI？
- 是否存在重复实现？
- 是否可解释、可复算、可测试？

### 架构

- 是否符合当前阶段目标？
- 是否影响种子用户试用版本冻结？
- 是否扩大了不可迁移的技术债？
- 是否为了长期方向过早进行大规模重构？
- 是否需要记录 Architecture Decision？

### 用户信任

- 数据来源和时间是否清楚？
- 缺失、过期、手动和估算状态是否明确？
- 错误是否会被静默吞掉？
- 产品说明与真实数据流是否一致？

## 20. Final Principle

Asset Trail 的工程目标是：

```text
快速验证
    +
数据可信
    +
边界清晰
    +
正确架构
    +
长期维护
    +
持续演进
```

应避免：

```text
快速堆功能
    +
前端硬编码
    +
重复业务逻辑
    +
数据来源不清
    +
不可迁移的临时结构
    +
后期集中重构
```

最终原则：

> 当前阶段优先保证种子用户试用版本稳定、可信和可验证；长期通过清晰的数据归属、领域模型、服务边界和渐进式迁移，使系统能够自然演进到多用户、多设备和云同步架构，而不是以未来架构为理由扩大当前 MVP 范围。

---

## 21. Current Stage Engineering Constraints

当前种子用户试用阶段，禁止主动执行：

- 大规模 React/TypeScript 重构；
- 私人资产数据云端迁移；
- 引入复杂基础设施；
- 替换已有核心流程；
- 为未来可能需求提前设计复杂系统。

除非同时满足以下条件：

1. 产品负责人明确确认；
2. 当前需求无法通过现有架构完成；
3. 已完成影响评估。
