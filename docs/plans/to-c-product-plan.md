# 资产轨迹 To C 产品规划

## 产品定位

资产轨迹面向个人投资者，定位为资产记录与投资复盘工具。

核心价值不是提供投资建议或实时行情，而是帮助用户管理多账户、多资产、多币种组合，记录投资决策，理解收益来源，并持续复盘自己的投资行为。

建议产品定位：

> 资产轨迹是一款面向个人投资者的资产记录和投资复盘工具，帮助用户更好地管理多账户、多资产、多币种组合，在持续复盘中提升投资能力和资产管理能力。

## MVP 功能范围

第一版上线应优先完成核心闭环，避免过早做复杂社区、实时行情和自动连接券商。

必须包含：

- 用户注册和登录
- 资产创建
- 持仓列表
- 历史持仓
- 账户总览
- 资产类别分布
- 账户曲线
- 单资产历史价格曲线
- 笔记
- 本地或云端数据备份
- 导入和导出
- 隐私说明和免责声明

暂不建议优先做：

- 实时社区
- 自动交易
- 投资推荐
- 收益承诺
- 复杂行情系统
- 自动连接券商 API

## 单资产历史价格曲线

用户创建资产时，资产的开仓价格和开仓日期可能来自历史上的某一天，而不是最近交易日。为了帮助用户理解单个资产从开仓到当前的价格变化，应支持查看单资产历史价格曲线。

MVP 第一版目标：

- 用户在创建资产时填写或选择资产代码、开仓日期、开仓价格。
- 系统根据资产代码获取最近三个月的历史收盘价、ETF 日线收盘价或基金净值。
- 在资产详情或持仓列表中展示该资产从开仓日到最新交易日的价格变化曲线。
- 每个资产的收益以最新价格或最新净值和开仓价或开仓净值计算。收益金额 = `(最新价格或净值 - 开仓价格或净值) × 持仓数量`；收益率 = `(最新价格或净值 - 开仓价格或净值) / 开仓价格或净值`。
- 曲线使用日频收盘价即可，不需要分钟级、盘口级或实时价格。
- 若开仓日早于最近三个月，第一版可以只展示最近三个月走势，并在图表中标注用户录入的开仓价格和开仓日期。
- 若开仓日处于最近三个月内，则曲线从开仓日开始展示到最新收盘价。

首版数据覆盖范围：

- A 股：沪深 300 全部成分股，不只保存指数本身。
- 美股：纳斯达克 100 全部成分股，不只保存指数本身；若同一公司存在多类证券代码，按证券代码保存。
- 港股：恒生科技指数全部成分股，不只保存指数本身，按指数成分股全集获取历史日收盘价。
- ETF：沪深交易所上市 ETF 和美股 ETF 白名单，首版优先覆盖宽基、行业和跨境 ETF，后续按用户搜索和真实持仓逐步扩展。
- 基金：国内公募开放式基金，首版优先支持用户搜索或手动录入后拉取净值，不默认加载全量基金。
- 价格类型：股票和 ETF 使用日收盘价；场外基金使用单位净值，展示收益曲线时可同时保存累计净值和复权单位净值。
- 时间范围：最近三个月。

暂不覆盖：

- 港股全市场历史价格，首版仅覆盖恒生科技指数相关股票。
- 全量 A 股和全量美股。
- 全量 ETF 和全量公募基金的首日上线覆盖。
- 实时行情。
- 盘中价格。
- 复权价格的复杂切换。
- 分红、拆股、配股等公司行为的完整处理。
- 基金持仓明细、基金经理变更、费率变更等基金基本面信息的完整覆盖。

数据设计建议：

- 建立资产代码表，区分资产类型、市场、交易所、代码、名称、币种、是否可交易和是否需要净值估算。
- 建立指数宇宙表和指数成分关系表，保存沪深 300、恒生科技和纳斯达克 100 的成分股全集、来源、抓取时间和生效区间。
- 建立历史价格表，保存交易日、收盘价、数据来源和更新时间。
- 建立基金净值表，保存净值日期、公告日期、单位净值、累计净值、复权单位净值、数据来源和更新时间。
- 建立数据任务表，记录每次历史回补和每日更新的任务状态、来源、起止日期、成功数量、失败原因和重试次数。
- 历史价格应做缓存，避免每次打开页面都请求外部行情源。
- 同一资产代码只保存一份价格数据，多个用户持仓复用同一份历史价格。
- 价格数据和用户持仓数据分离，便于后续扩展更多市场。

数据获取方案调研：

- A 股股票、A 股 ETF 和国内公募基金建议优先评估 Tushare Pro 或同类有授权的数据服务。Tushare 已提供公募基金列表 `fund_basic`、公募基金净值 `fund_nav`、场内基金日线行情 `fund_daily` 等接口，适合统一拉取基金基础信息、ETF 日线和场外基金净值。
- 港股建议首版只接入恒生科技指数成分股白名单，日线价格可评估 Tushare `hk_daily` 或其他具备港股授权的行情服务。成分股名单需要定期同步，避免指数调样后继续拉取过期标的或漏掉新增标的。
- 沪深交易所官网可作为 ETF 列表、基金行情和交易日信息的人工核验来源，不建议在生产环境依赖网页抓取作为主数据源。
- 美股股票和美股 ETF 可评估 Alpha Vantage、EODHD、Polygon 等海外行情服务。MVP 只需要日频历史收盘价，优先选择支持 ETF、历史 EOD、调整收盘价、调用额度管理和商业授权明确的数据源。
- AkShare、东方财富、天天基金等公开数据抓取方式可用于内部原型验证，但上线前必须确认授权、稳定性、限频和缓存策略，不应作为面向用户的唯一生产数据源。
- 基金和 ETF 需要分开建模：ETF 是交易所交易品种，使用交易日收盘价；场外基金以基金公司公告净值为准，通常存在 T+1 或更晚披露，不应展示为实时价格。

- Tushare 公募基金列表：<https://tushare.pro/document/1?doc_id=19>
- Tushare 公募基金净值：<https://tushare.pro/document/2?doc_id=119>
- Tushare ETF 日线行情：<https://tushare.pro/document/2?doc_id=127>
- Tushare 港股日线行情：<https://www.tushare.pro/document/2?doc_id=192>
- 上交所基金列表：<https://etf.sse.com.cn/fundlist/>
- 深交所基金列表：<https://www.szse.cn/market/fund/list/stockFundList/index.html>
- Alpha Vantage API 文档：<https://www.alphavantage.co/documentation/>
- EODHD 历史 EOD API：<https://eodhd.com/lp/historical-eod-api>

历史数据回补方案：

- 首次上线先为首版白名单资产批量回补最近三个月数据，股票和 ETF 按交易日收盘价入库，场外基金按净值日期入库。
- 用户新增资产时，如果资产不在缓存中，后端异步创建回补任务，拉取该资产最近三个月数据；若用户开仓日早于三个月，MVP 仍只回补三个月，并保留用户录入的开仓价或开仓净值。
- 所有外部返回的价格、净值、成交量和涨跌幅先进入 staging 记录，完成字段校验、日期去重、交易日校验和 Decimal 转换后再写入正式表。
- 同一资产、同一交易日或净值日期使用唯一约束去重；后续数据源修正历史值时，保留更新前后的审计记录和来源时间。
- 历史回补失败时不阻塞用户保存资产，应显示“行情缺失 / 净值缺失”状态，并允许用户继续使用手动录入价格。

每日最新数据更新方案：

- 按市场日历调度更新任务。A 股股票和 ETF 在交易日收盘后执行，国内场外基金在净值披露窗口后执行，并在次日补跑一次处理迟到净值。
- 美股股票和 ETF 在美股交易日收盘后按 UTC 次日早晨执行，避免把盘中延迟报价误认为最终收盘价。
- 每日任务按资产类型拆分队列：A 股股票、A 股 ETF、恒生科技港股、美股股票、美股 ETF、场外基金净值分别执行，避免单一数据源失败影响全部资产。
- 最新价发布前做质量检查：日期不能早于已发布最新日期，价格或净值必须为正数，异常涨跌幅需要标记，缺失数据必须写入失败原因。
- 对外展示时明确标注价格或净值日期、数据来源和更新时间。对基金净值延迟、停牌、节假日、数据源失败等情况给出可理解的状态。
- 数据更新只负责记录和分析，不触发任何交易建议、收益承诺或个性化推荐。

数据源注意事项：

- 上线前需要确认行情数据源的授权范围、调用频率和商业使用限制。
- 页面中应说明价格数据可能延迟，仅用于记录和分析。
- 若数据缺失，应允许用户手动录入或继续使用用户录入价格。
- 对于代码重名、退市、停牌、无交易日等情况，需要有明确的空状态和错误提示。

## 技术重构方向

当前原生 JS 适合原型验证。若要上线为 To C 产品，建议重构为更可维护的工程结构。

推荐技术栈：

- 前端：React + Vite + TypeScript
- 样式：Tailwind CSS 或 CSS Modules
- 后端：Node.js / NestJS / Fastify
- 数据库：PostgreSQL
- ORM：Prisma
- 登录：邮箱验证码、OAuth，后续支持 2FA
- 部署：Vercel / Cloudflare Pages + 后端服务，或 AWS ECS / Lightsail
- 数据导入导出：CSV / JSON

建议目录结构：

```text
apps/
  web/
  api/
packages/
  domain/
  ui/
  shared/
prisma/
docs/
```

## 数据模型规划

资产管理产品的核心是数据模型。需要提前区分资产、持仓、交易和估值快照。

核心表：

- `users`
- `accounts`
- `assets`
- `positions`
- `transactions`
- `valuation_snapshots`
- `notes`
- `closed_positions`
- `settings`
- `imports`
- `audit_logs`

关键概念：

- 资产：例如 AAPL、BTC、现金。
- 持仓：某个账户持有某个资产的数量和成本。
- 交易：买入、卖出、分红、转入、转出。
- 快照：某个时间点的组合价值。

不建议长期只依赖“当前价格 + 数量”计算全部结果，否则后续扩展交易流水、历史收益和复盘分析会很困难。

## 金融合规边界

To C 金融相关产品需要严格控制表达。

必须避免：

- 推荐买卖
- 承诺收益
- 暗示跟单
- 排名诱导
- 社区带单
- 未授权展示投资建议

页面、用户协议和隐私政策应明确：

- 产品仅用于记录和分析
- 不构成投资建议
- 数据由用户自行录入
- 收益计算可能存在误差
- 用户自行承担投资决策风险

社区上线前应先设计审核机制和举报机制。

## 隐私与安全

隐私可以作为资产轨迹的重要差异化方向。

建议强调：

- 用户可以选择只在本地保存数据
- 云同步可选
- 支持导出备份
- 不强制连接券商或银行
- 敏感数据加密
- 删除账户时彻底删除数据

基础安全要求：

- HTTPS
- 密码哈希
- CSRF / XSS 防护
- API 权限校验
- 数据库备份
- 操作日志
- 速率限制
- 隐私政策
- 用户协议

## 商业化规划

建议优先考虑轻量订阅制。

免费版：

- 本地记录
- 少量资产
- 基础图表
- 基础笔记

Pro 版：

- 云同步
- 多账户
- 导入导出
- 高级复盘
- 汇率影响分析
- 目标追踪
- 更多历史快照
- 数据备份

早期不建议依赖广告。资产数据属于高信任场景，广告会削弱用户信任。

## 上线优先级

建议按以下顺序推进：

1. 产品文档、PRD 和领域数据模型：明确资产、账户、交易、持仓、估值快照、价格、基金净值、数据任务、笔记、社区内容和审计日志边界。
2. React + TypeScript 工程化重构：拆分当前原型，建立 `domain/`、`features/`、`server/`、`db/` 等模块边界，并保留现有金融计算测试。
3. 金融计算与归因内核：优先固化金额、数量、汇率、费用、税费、分红/利息、手动调整和未归因差异的纯函数与单元测试。
4. 数据库和 API 基础：落地 PostgreSQL、migration、服务端校验、权限检查、审计字段和错误处理。
5. 用户系统和隐私设置：完成注册登录、会话、权限、最小化数据访问、账户删除、导出和本地/云同步选择。
6. 资产、账户、交易、持仓和历史持仓：完成录入、编辑、清仓、交易流水、附件记录、导入预览、校验和去重。
7. 首版行情 / 净值数据管线：覆盖沪深 300、纳斯达克 100、恒生科技、ETF 白名单和国内公募基金净值，支持历史回补、每日更新、失败重试和数据来源展示。
8. 账户快照、组合曲线和单资产历史价格曲线：支持最近三个月曲线、价格 / 净值日期、数据缺失状态和用户手动价格兜底。
9. 资产变化归因分析：解释组合价值从起始日到结束日的变化，确保贡献项加总后能对齐总变化。
10. 投资复盘笔记：支持草稿、私有、发布状态，关联资产、账户、交易、时间区间和标签。
11. 导入导出和备份恢复：支持 CSV / JSON 导出、导入预览、校验、重复提交防护和恢复流程。
12. 合规与安全文案：完善隐私政策、用户协议、免责声明、风险提示、外部数据授权说明和社区表达边界。
13. 部署上线和监控：完成生产环境配置、备份、日志、速率限制、错误监控和数据任务监控。
14. 支付订阅：在核心记录、分析、数据同步和备份体验稳定后，再接入轻量订阅制。
15. 社区：最后上线社区发布、评论、点赞、举报、审核、隐藏、删除、封禁和申诉记录，确保与用户资产数据隔离。

## 近期行动建议

当前最值得优先做：

- 基于 `docs/product/mvp-prd.md`、`docs/data/domain-model.md` 和 `db/schema.sql` 启动 React + TypeScript 工程化重构。
- 拆分 `src/app.js`，把页面、feature、领域计算、数据任务和导入导出分离。
- 建立 PostgreSQL migration 和后端 API 骨架，对齐 `docs/data/api-contract.md`。
- 把现有本地登录升级为服务端认证、权限检查和审计日志。
- 将当前本地导入 / 导出升级为导入预览、校验、去重和事务确认流程。
- 先用公开数据源跑 `npm run data:backfill` 回补三个月，再用 `npm run data:daily` 做每日更新；上线前再评估是否切换为授权商业数据源。
- 补充用户协议、隐私政策、免责声明和外部数据授权说明草稿。

建议先不要急着做社区，也不要急着接实时行情。先把“个人资产记录 + 复盘 + 隐私”这个闭环做扎实。
